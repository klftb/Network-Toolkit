本文档详细记录了最近所有关于 Network Toolkit 软件中出现的公网 IP 检测偏移、Windows 下 Wi-Fi 扫描异常、信道识别和浏览器基站辅助定位等问题的反馈、原因分析以及已实施的完全修复方案。

## 一、 最新问题汇总与核心修复技术

在近期的调试与使用中，主要对以下两部分网络核心功能进行了全面排障和深度完善：

### 1. 公网 IP 探测源重构与解析稳定性修复
* **用户问题**: 之前设定的国内公网 IP 测试节点（如 `ip.tool.chinaz.com`）出现无法解析、返回错误或高延迟等异常，导致获取本地网络出口 IP 失败。
* **原因分析**: 
  1. 许多国内 IP 测试地址更新了反爬或改变了网页 HTML 结构，使得针对特定 `id` 的正则解析规则直接失效。
  2. 某些节点对请求的 HTTP 头部有严格校验，未加 `User-Agent` 的 Node `fetch` 极易遭到阻断（例如直接返回 `403 Forbidden` 状态码）。
* **解决方案**:
  1. 将国内公网 IP 节点地址换为了 **`www.ip38.com`**。
  2. 在 `server.ts` 中，为发起 fetch 请求增加了完整的浏览器 User-Agent 伪装头部：
     `'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; ...)'`
  3. 针对 `www.ip38.com` 的响应 HTML，重构了多路正则回退解析规则。基于页面规律，直接捕获带有 `.htm` 的 IP 落地页链接值：
     `const match = text.match(/<a[^>]*>(\d+\.\d+\.\d+\.\d+)<\/a>/) || text.match(/(\d+\.\d+\.\d+\.\d+)\.htm/);`
     在提取出 IP 字符串（如 `34.34.226.212`）后直接返回，完美修复了公网 IP 出口测速检测异常。

### 2. Windows 系统 Wi-Fi 扫描及 2.4G/5G 频段丢失、不显示 5G 信号问题修复
* **用户问题**: Windows 部分中文字符集环境下使用 Wi-Fi 扫描工具，扫描列表为空；或者在未建立 Wi-Fi 连接的状态下无法识别频段信息，且不显示 5G Wi-Fi。
* **原因分析**:
  1. **命令编码乱码**: `node-wifi` 依靠调用 `netsh wlan show networks mode=bssid` 来获取 Windows 系统的 Wi-Fi。在中文系统下，系统默认的活动代码页是非 UTF-8 编码（例如 GBK/CP936），若直接按 UTF-8 或者简单的二进制流转换，汉字（如“信号”、“身份验证”、“信道”）会产生阻断。原有的 `chcp 65001` 在部分没有完整多语言环境的环境下执行命令行会抛出解析异常，导致扫描中途崩毁返回空列表。
  2. **多语言关键词未覆盖**: `node-wifi` 库本身的解析代码极度僵硬，只识别了英文关键字。它无法精确提取中文 netsh 输出的“信道”、“信号”等结构化属性。
  3. **非连接状态时频段缺失**: 部分网卡在没有主动连接时，`netsh` 返回的信息可能丢失具体的 MHz 频率属性。原程序逻辑直接卡在 `net.frequency < 3000` 导致全部显示为 2.4GHz（或是直接丢失），使 5GHz 网频被滤除。
* **解决方案**:
  1. **完全替代并重写 Windows 扫描解析（不依赖 node-wifi 解析器）**:
     我们在后端中通过 `execSync('netsh wlan show networks mode=bssid')` 抓取最原始的 Buffer。然后引入了强大的外部多字符编码库 **`iconv-lite`**。检测其是否包含特定乱码或默认采用 `cp936` 对 Buffer 进行高精度无损解码。
  2. **中英文多路不规则匹配提取**:
     重构后直接捕获每一条 `SSID` 块和多路的 `BSSID` 块，能够识别全量 Windows 输出。解析子层中全面适配了中文和英文词汇：
     * **信道**: `key.includes('channel') || key.includes('信道') || key.includes('信 道') || key.includes('频段')`
     * **信号强度**: `key.includes('signal') || key.includes('信号')`
     * **无线电/频宽类型**: `key.includes('radio') || key.includes('无线') || key.includes('类型')`
     * **速度/速率**: `key.includes('rate') || key.includes('速率')`
     * **身份验证**: `key.includes('auth') || key.includes('验证') || key.includes('认证')`
  3. **前/后端智能频率推导计算（修复非连接缺失）**:
     不管是后端还是前端列表组件，凡是未自带 Frequency 的 Wi-Fi 数据，都通过信道号码（Channel）进行无缝换算：
     * 如果信道在 `1` 到 `14` 之间，则自动设定为 2.4 GHz 频段，对应的中心频率由公式 `2407 + channel * 5 (MHz)` 推导。
     * 如果信道大于 `14` 最多到 `165`（例如 `36, 40, 149, 157` 等），自动判定为 5 GHz 频段，对应中心频率由公式 `5000 + channel * 5 (MHz)` 指代。
     该机制弥补了 Windows 网卡数据空缺，无论是 2.4G 还是 5G 频道全都恢复无缝图形渲染。
  4. **引入浏览器 Geolocation API 实现高精度基站定位参考**:
     我们在前端 Wi-Fi 工具箱中增加了全新的“获取基站定位”按钮。使用符合标准的 `navigator.geolocation` 接口并指定 `{ enableHighAccuracy: true, timeout: 15000 }`。可以一键捕获到当前终端的经纬度座标、定位精确度半径以及捕获秒级时间戳，以便于网络测试人员在离线测试和排查 Wi-Fi 信号覆盖、基站信号强弱时作为精确的现场物理位置参考档案。
